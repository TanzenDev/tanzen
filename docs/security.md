# Tanzen Security Plan

This document covers three things: how to manage security during development, the security baseline for the OSS core, and the enterprise security roadmap from first OIDC to IL5 certification.

---

## 1. Development Security Modes

Tanzen has two intended dev modes. The right one depends on whether you are working on security itself.

### 1.1 Frictionless mode (default)

Most contributors are not working on auth, RBAC, or SSO. For them, security should be invisible.

**How it works today:** When `CLERK_SECRET_KEY` is absent from `server/.env`, the server boots with `DevAuthProvider`, which grants every request `role: "admin"` under the identity `TANZEN_DEV_USER` (default: `"dev-user"`). No tokens, no login, no browser extension needed. The React app makes unauthenticated API calls that the server accepts unconditionally.

**What to preserve:** This mode should always work with a plain `bun run dev` and a fresh `.env` generated by `dev-env.sh`. No security-related env var should ever be required to start the server in development.

**Risks to contain:** DevAuthProvider must only activate when `NODE_ENV !== "production"`. Add a startup guard so that if someone accidentally deploys a server without credentials, the process refuses to start rather than silently granting admin to everyone:

```typescript
// server/src/server/auth.ts
if (!CLERK_SECRET_KEY && process.env["NODE_ENV"] === "production") {
  throw new Error("CLERK_SECRET_KEY must be set in production");
}
```

Add a console warning (not an error) when DevAuthProvider activates:

```typescript
console.warn("[auth] DevAuthProvider active — all requests granted admin. Set CLERK_SECRET_KEY or OIDC_ISSUER to enable real auth.");
```

### 1.2 Security debug mode

Developers working on auth, RBAC, multi-tenancy, or the enterprise plugins need to run the full auth stack locally without a staging cluster.

**Role impersonation:** The simplest tool is a `TANZEN_DEV_ROLE` env var that DevAuthProvider respects. This lets a developer test the viewer or reviewer code path without touching JWT machinery:

```bash
TANZEN_DEV_ROLE=viewer bun run dev         # viewer perspective
TANZEN_DEV_ROLE=reviewer bun run dev       # reviewer perspective
TANZEN_DEV_ROLE=editor bun run dev         # enterprise editor role
```

Implementation — update `DevAuthProvider`:

```typescript
class DevAuthProvider implements AuthProvider {
  async authenticate(_req: Request, _c: Context): Promise<AuthUser | null> {
    const role = process.env["TANZEN_DEV_ROLE"] ?? "admin";
    return { userId: DEV_USER_ID, role };
  }
}
```

**Local Clerk:** Set `CLERK_SECRET_KEY` and `CLERK_PUBLISHABLE_KEY` from a Clerk development instance. The Clerk dev instance issues real JWTs against a local user database. Use `@clerk/testing` helpers to issue tokens programmatically in integration tests.

**Local OIDC (enterprise):** Run a local Keycloak or Dex container. `dev-env.sh` should grow an optional `--with-oidc` flag that starts a Docker Keycloak container, imports a realm fixture, and writes the `OIDC_ISSUER`, `OIDC_CLIENT_ID`, and `OIDC_CLIENT_SECRET` vars into `server/.env`. This gives a fully local OIDC round-trip without any cloud dependency.

Sample Keycloak compose fragment (`infra/dev/oidc.yaml`):

```yaml
services:
  keycloak:
    image: quay.io/keycloak/keycloak:24
    command: start-dev --import-realm
    environment:
      KEYCLOAK_ADMIN: admin
      KEYCLOAK_ADMIN_PASSWORD: admin
    volumes:
      - ./keycloak-realm.json:/opt/keycloak/data/import/tanzen.json
    ports:
      - "8180:8080"
```

**Testing RBAC failures:** The existing `hardening.test.ts` matrix covers this at the integration level. For manual debugging, pair `TANZEN_DEV_ROLE` with `curl` or the browser devtools network tab to observe 403 responses.

**Security test isolation:** The `hardening.test.ts` suite should be runnable standalone without a live cluster:

```bash
bun test server/tests/server/hardening.test.ts
```

This is already the case today; keep it that way. Add a `bun run test:security` script alias so security-focused contributors have a clear entry point.

## 2. OSS Core Security — Implementation Plan

The OSS core (`server/`, `app/`) has a functional auth foundation but several gaps that need closing before the 1.0 release. These are concrete, bounded changes.

### 2.1 HTTP security headers

**Gap:** No security headers are set. Browsers receive no CSP, no HSTS, no clickjacking protection.

**Fix:** Add `hono/secure-headers` middleware to `server/src/server/index.ts` before the CORS middleware. Hono ships this built-in.

```typescript
import { secureHeaders } from "hono/secure-headers";

app.use("*", secureHeaders());
app.use("*", cors({ ... }));
```

Default `secureHeaders()` sets: `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy: strict-origin-when-cross-origin`, `Permissions-Policy`, and `X-XSS-Protection: 0` (correct for modern browsers). HSTS (`Strict-Transport-Security`) requires TLS termination upstream — add it only behind the Ingress or a dedicated middleware that checks `X-Forwarded-Proto`.

**CSP for the React app:** The Vite build (`app/vite.config.ts`) should generate a `<meta http-equiv="Content-Security-Policy">` tag at build time. Start permissive and tighten incrementally:

```
default-src 'self';
script-src 'self' 'nonce-{VITE_CSP_NONCE}';
style-src 'self' 'unsafe-inline';
connect-src 'self' https://api.clerk.com;
img-src 'self' data:;
frame-ancestors 'none';
```

Add `VITE_CSP_NONCE` generation to the Vite plugin config. The `'unsafe-inline'` for styles is a concession to Tailwind's inline styles; replace with a hash allow-list once the style set is stable.

### 2.2 CORS hardening

**Gap:** `ALLOWED_ORIGINS` defaults to `"*"`, which is permissive and reflects no intent.

**Fix:** Change the default to be environment-aware:

```typescript
function defaultOrigins(): string | string[] {
  if (process.env["NODE_ENV"] === "production") {
    throw new Error("ALLOWED_ORIGINS must be set in production");
  }
  return ["http://localhost:5173", "http://localhost:3000"];
}

app.use("*", cors({
  origin: process.env["ALLOWED_ORIGINS"]?.split(",") ?? defaultOrigins(),
  allowHeaders: ["Authorization", "Content-Type"],
  maxAge: 600,
}));
```

This keeps local dev working without any config change while forcing an explicit decision in production deployments. The Helm chart (`infra/charts/tanzen`) should add `ALLOWED_ORIGINS` to its values and populate it from the ingress host.

### 2.3 Request body size limits

**Gap:** No maximum request size. A client can stream an arbitrarily large body to any endpoint.

**Fix:** Hono does not ship a body-size middleware, but Bun's HTTP server accepts `maxRequestBodySize`. Set this in `server/src/server/index.ts` export:

```typescript
export default {
  port,
  fetch: app.fetch,
  idleTimeout: 0,
  maxRequestBodySize: 10 * 1024 * 1024, // 10 MB
};
```

For the DSL compile endpoint (`POST /api/workflows`), add an explicit 500 KB guard since DSL source should never be larger than that. Return `413 Payload Too Large` with a descriptive message.

### 2.4 Input validation — centralise with Zod

**Gap:** Validation is currently spread across individual route handlers as ad-hoc regex checks and conditional statements. This makes it hard to audit and easy to miss a new endpoint.

**Fix:** Introduce Zod schemas colocated with each route file. Use a shared `validate()` helper that returns a typed value or throws a 422:

```typescript
// server/src/server/validate.ts
import { ZodSchema } from "zod";
import type { Context } from "hono";

export async function parseBody<T>(c: Context, schema: ZodSchema<T>): Promise<T> {
  const body = await c.req.json().catch(() => null);
  const result = schema.safeParse(body);
  if (!result.success) {
    throw new HTTPException(422, { message: result.error.message });
  }
  return result.data;
}
```

Route files then look like:

```typescript
const CreateAgentSchema = z.object({
  name: z.string().regex(/^[a-z][a-z0-9-]*$/),
  model: z.enum(ALLOWED_MODELS),
  system_prompt: z.string().max(32_768),
  max_tokens: z.number().int().min(1).max(128_000),
  temperature: z.number().min(0).max(2),
});

agentRoutes.post("/", requireRole("admin", "author"), async (c) => {
  const body = await parseBody(c, CreateAgentSchema);
  ...
});
```

This work can be done incrementally, one route file at a time, without breaking anything.

### 2.5 Rate limiting — Redis persistence for production

**Gap:** The current `rateLimit()` implementation uses an in-process `Map`. In a multi-replica Kubernetes deployment, each pod has an independent counter. A user can exceed the per-user limit by hitting different pods.

**Fix:** Make the rate limiter storage backend pluggable. The default (in-process Map) stays for single-instance dev. Production deployments inject a Redis-backed store:

```typescript
// server/src/server/ratelimit.ts
export interface RateLimitStore {
  increment(key: string, windowMs: number): Promise<{ count: number; resetAt: number }>;
}

export function rateLimit(opts: { windowMs: number; max: number; keyFn?: KeyFn; store?: RateLimitStore }) {
  const store = opts.store ?? new InMemoryStore();
  ...
}
```

The commercial build passes a Redis store via the plugin's `onStartup` hook. Document the Redis store as the required production configuration in the deployment guide.

### 2.6 Dev bypass production guard

Already described in §1.1. File: `server/src/server/auth.ts`. Two changes:

1. Throw at startup if `DevAuthProvider` would activate in production.
2. Log a warning in non-production environments when `DevAuthProvider` activates.

### 2.7 `/metrics` and `/health` endpoints

**Gap:** Both are unauthenticated. `/metrics` exposes process and application metrics (request counts, queue depths, memory). In a public deployment this leaks operational information.

**Fix:** Guard `/metrics` with a static bearer token (`METRICS_TOKEN` env var). Kubernetes kube-prometheus scrapeConfigs support `bearerToken`:

```typescript
app.get("/metrics", async (c) => {
  const token = process.env["METRICS_TOKEN"];
  if (token) {
    const auth = c.req.header("Authorization");
    if (auth !== `Bearer ${token}`) return c.json({ error: "Unauthorized" }, 401);
  }
  ...
});
```

`/health` is legitimately unauthenticated (used by liveness probes); keep it open but return minimal information (just `{ ok: true }`).

### 2.8 Frontend auth stub

**Gap:** The React app mounts all routes without auth guards. In OSS this is intentional since the server enforces auth, but the app has no hook for the enterprise layer to add redirect-to-login behavior.

**Fix:** Add a `ProtectedRoute` wrapper component that calls `useSlot("auth.guard")`. In OSS the slot returns null and the route renders normally. The enterprise build fills the slot with a redirect guard and token refresh logic.

```tsx
// app/src/components/ProtectedRoute.tsx
import { useSlot } from "../extensions/registry";

export function ProtectedRoute({ children }: { children: React.ReactNode }) {
  const Guard = useSlot("auth.guard");
  if (Guard) return <Guard>{children}</Guard>;
  return <>{children}</>;
}
```

Wrap all app routes in `App.tsx` with `<ProtectedRoute>`.

### 2.9 Secrets route — remove TLS skip comment

`server/src/server/routes/secrets.ts` has a comment acknowledging that the kubectl proxy bypasses TLS. This is correct behavior but the comment should be a proper note explaining the design decision (Bun's native fetch ignores custom HTTPS agents, so the proxy is intentional), not a passing reference that might alarm a security auditor.

### 2.10 Security test baseline

`server/tests/server/hardening.test.ts` is the authoritative record of security behaviors. It must:

- Run in CI on every PR (verify this is true — add to the CI workflow if not).
- Cover every new route added to the server (enforce via a lint rule or a test that counts registered routes and compares against tested routes).
- Include a test that `DevAuthProvider` is never active when `NODE_ENV=production` (mock the env and assert the startup error).

Add `bun run test:security` to `package.json`:

```json
"test:security": "bun test server/tests/server/hardening.test.ts"
```

## 3. Enterprise Security Roadmap

The enterprise repo (`tanzen-enterprise`) already has working implementations of OIDC authentication, a four-role RBAC model, org-based multi-tenancy, and an audit log. The roadmap starts from fixing known gaps and then extends toward compliance.

### Phase 0: Bug fixes (do now, before any production traffic)

These are bugs that exist in the current codebase.

**0.1 Unguarded org list endpoint**

`GET /api/tenancy/orgs` in `server/src/plugins/tenancy/index.ts` (line ~23) returns all organizations to any authenticated user. This allows any user to enumerate every tenant in the system.

Fix: add `requirePermission("orgs:read")` — which the RBAC matrix already restricts to admin — as the first middleware on this route.

**0.2 Token stored in localStorage**

`app/src/features/sso/OidcCallback.tsx` writes the OIDC access token to `localStorage`. Any script running on the page (including injected third-party scripts or an XSS payload) can read it.

Fix: Change the OIDC callback to exchange the authorization code entirely server-side, then issue an `HttpOnly; Secure; SameSite=Strict` session cookie. The browser never holds a raw token. This requires a thin session table (or Redis session store) server-side, but eliminates the XSS token-theft vector entirely.

Intermediate fix (if the cookie refactor is not yet scoped): move from `localStorage` to `sessionStorage`. This does not stop XSS from reading it during the session but prevents token persistence across tabs and reduces the window for harvesting.

**0.3 Audit log does not capture failed mutations**

`server/src/plugins/audit/middleware.ts` only logs 2xx responses. A brute-force attempt or an authorization bypass attempt generates no audit record.

Fix: Log all mutations (POST/PUT/PATCH/DELETE) regardless of status code. Include the HTTP status in the log entry (it's already in the schema). For 401/403 responses, set `actor_id` to the extracted JWT subject if available, or `"anonymous"` otherwise.

**0.4 No permission check on audit log routes**

`GET /api/audit/entries` is reachable by any authenticated user. The audit log contains actor IDs, resource IDs, and timestamps for every mutation in the system.

Fix: add `requirePermission("audit:read")` — already defined in the permission matrix as admin-only — to the audit routes.

### Phase 1: OIDC hardening (2–4 weeks)

**1.1 PKCE for the authorization code flow**

The current OIDC flow (`server/src/plugins/oidc/routes.ts`) uses the authorization code flow without PKCE. PKCE (RFC 7636) protects against authorization code interception and is required for public clients (SPAs, native apps). Even for confidential clients, PKCE is now recommended by RFC 9700.

Implementation: generate `code_verifier` (32 random bytes, base64url-encoded) in `/api/oidc/login`, store it in an `HttpOnly` cookie alongside the nonce, pass `code_challenge` and `code_challenge_method=S256` in the authorization redirect, and send `code_verifier` in the token exchange at `/api/oidc/token`.

**1.2 Token introspection / revocation**

JWTs are stateless. A revoked token (e.g., after a user is removed from an org) remains valid until expiry. For workflows that require immediate revocation (terminated employee, security incident), add token introspection against the OIDC provider on sensitive operations, or maintain a local revocation list in Redis.

Minimum viable: on every request, check if the token's `jti` (JWT ID) is in a Redis block-list. Add `POST /api/oidc/revoke` (admin-only) to add a `jti` to the block-list. Set the Redis key TTL to the token's remaining lifetime.

**1.3 OIDC provider configuration guide**

Write `docs/oidc-setup.md` covering Keycloak, Auth0, Okta, and Microsoft Entra. For each: how to create the client, what claims to configure (`tanzen_role`, `org_id`), redirect URI patterns, and how to map IdP groups to Tanzen roles. This is a developer-hours multiplier — teams blocked on OIDC config cost more than the doc takes to write.

**1.4 Rate limits on auth endpoints**

`/api/oidc/login` and `/api/oidc/token` are not covered by the per-user rate limiter (the user isn't authenticated yet). Apply a strict IP-based rate limit: 10 requests/minute on `/api/oidc/login`, 20 on `/api/oidc/token`.

**1.5 Session management**

If the cookie-based session approach from §0.2 is adopted, add:
- Session expiry matching the OIDC token lifetime
- Silent refresh: before making an API call, check if the token expires within 60 seconds and refresh proactively
- Logout: call `/api/oidc/logout`, clear the session cookie, redirect to IdP end-session endpoint

### Phase 2: RBAC and multi-tenancy hardening (4–8 weeks)

**2.1 Row-level security**

The current tenancy model attaches `orgId` to the request context (from the OIDC claim), but downstream route handlers are responsible for filtering queries by `org_id`. If a handler forgets the filter, it leaks cross-tenant data.

Fix: add Postgres row-level security (RLS) policies on all tenant-scoped tables. Set `app.current_org_id` as a Postgres session variable at the start of each request and write RLS policies that check it:

```sql
ALTER TABLE workflows ENABLE ROW LEVEL SECURITY;
CREATE POLICY workflow_tenant_isolation ON workflows
  USING (org_id = current_setting('app.current_org_id'));
```

The tenancy middleware sets this variable after establishing the org context. This makes tenant isolation a database-enforced invariant rather than a code convention.

**2.2 Tenancy isolation tests**

Write integration tests that:
- Create two orgs (org_a, org_b) and a user in each
- Assert that org_a's user cannot read, write, or delete org_b's workflows, agents, runs, or secrets
- Assert that org_a's admin cannot list or modify org_b's membership

These tests should run in CI against a real Postgres instance (the existing test infrastructure already supports this).

**2.3 Resource quotas per org**

Without quotas, a single org can exhaust cluster resources. Add a `quotas` column to the `organizations` table and enforce limits in route handlers:

```sql
ALTER TABLE organizations ADD COLUMN quotas JSONB DEFAULT '{"max_workflows": 100, "max_agents": 50, "max_runs_per_day": 1000}';
```

Enforce in the workflow and run creation routes before inserting. Return `429 Too Many Requests` with a descriptive body when a quota is exceeded.

**2.4 RBAC audit**

Every permission denial (403) should be logged to the audit table with the required permission and the user's actual role. This supports compliance investigations ("why couldn't user X do Y on date Z?") and helps detect privilege escalation attempts.

**2.5 Role assignment API**

Currently roles are assigned via the OIDC provider (the `tanzen_role` claim). Add a `POST /api/tenancy/orgs/:id/members/:userId/role` endpoint (admin-only) that:
- Updates `org_members.role` in Postgres
- Validates the new role against the allowed enum
- Logs the change to the audit table with the previous role and the actor making the change

This lets org admins manage roles within their org without requiring access to the IdP.

### Phase 3: SOC 2 Type II (sketch, 3–6 months)

SOC 2 Type II requires demonstrating that controls were operating continuously over an audit period (typically 6 months). The technical foundations are largely in place; the remaining work is policy, process, and evidence collection.

**Security controls to implement:**

- *Encryption at rest:* Verify that Postgres (CloudNativePG) and SeaweedFS volumes are encrypted at the storage layer. Document the encryption configuration in the deployment guide. For sensitive workflow artifacts in SeaweedFS, evaluate server-side encryption at the application layer (encrypt before upload, decrypt on read, key stored in K8s secrets or a KMS).

- *Encryption in transit:* Enforce TLS 1.3 minimum on all ingress. Add an Ingress annotation (`nginx.ingress.kubernetes.io/ssl-protocols: "TLSv1.3"`) and document it. Mutual TLS (mTLS) between internal services (worker → Temporal, server → Postgres) should be verified and documented.

- *Audit log retention:* Define a retention policy (90 days hot, 1 year cold). Add a Kubernetes CronJob that exports audit log rows older than 90 days to S3 and deletes them from Postgres. Write a restore procedure. For IL2+, the retention period may be mandated by NIST 800-53 AU-11.

- *Vulnerability management:* Add `trivy` image scanning to the CI pipeline for all container images. Add `govulncheck` for the Go CLI, `npm audit` for the app, and `uv audit` for the worker. Define a policy: critical CVEs patched within 14 days, high within 30 days.

- *Access review:* Quarterly review of all org members and their roles. Add an admin report endpoint (`GET /api/admin/access-review`) that returns a flat list of all users, orgs, and roles for audit export.

- *Incident response:* Write `docs/incident-response.md` covering: detection (alerting thresholds), classification (P1–P4), escalation path, communication template, and post-mortem format.

- *Penetration testing:* Engage an external firm for an annual black-box pentest. Fix all critical and high findings before the SOC 2 audit window begins.

- *Vendor management:* Document third-party services (Clerk, Temporal Cloud if used, any AI model providers) and their own compliance attestations.

**Policies to write:**

- Information security policy
- Access control policy  
- Change management policy (already implied by git/PR workflow; formalize it)
- Incident response policy
- Data retention and deletion policy
- Acceptable use policy

### Phase 4: IL5 (sketch, 12–24 months)

IL5 (Impact Level 5) covers Controlled Unclassified Information (CUI) at the highest sensitivity level. It requires FedRAMP High authorization as a prerequisite for cloud deployments, or an on-premises deployment meeting equivalent controls.

**Technical requirements delta from SOC 2:**

- *FIPS 140-3 cryptography:* All cryptographic operations must use FIPS-validated modules. This affects: TLS libraries (use BoringSSL/FIPS builds), JWT signing (verify the OIDC provider's FIPS posture), storage encryption (verify CloudNativePG and SeaweedFS FIPS modes). Go's `crypto/tls` can use FIPS-compliant backends with build tags (`GOEXPERIMENT=boringcrypto`).

- *PIV/CAC authentication:* Federal users authenticate with PIV cards (HSPD-12). Replace the OIDC password flow with a PIV-capable IdP (DoD PKI, ICAM). This requires the OIDC provider to support certificate-based authentication (CBA), which Keycloak supports via x509 authenticators.

- *Network isolation:* Deploy within a FedRAMP-authorized cloud region (AWS GovCloud, Azure Government) or an on-premises environment with DISA-approved network controls. The Cilium NetworkPolicies already in the Helm chart are the right starting point; extend them to implement a zero-trust architecture where every service-to-service call is explicitly allowed.

- *Data sovereignty:* All data at rest and in transit must remain within the authorization boundary. This requires: no calls to external AI provider APIs from within the boundary (deploy on-premises models or use approved GovCloud AI services), no external CDN for the frontend (self-host all assets), no third-party auth SaaS (run Keycloak within the boundary).

- *Continuous monitoring:* SIEM integration (Splunk, Elastic SIEM) consuming audit logs, Kubernetes audit logs, and network flow logs. Define alert rules for NIST 800-53 SI-4 (system monitoring) controls. This is where the audit log's structured format pays off — pipe it directly to the SIEM.

- *Personnel security:* Operators of IL5 systems require appropriate clearances or investigations. This is an organizational requirement, not a software one, but the RBAC model must support role assignment tied to clearance level.

- *Assessment and authorization (A&A):* Engage a Third Party Assessment Organization (3PAO) accredited by the FedRAMP PMO. The A&A process typically takes 6–12 months and requires a System Security Plan (SSP), Security Assessment Report (SAR), and Plan of Action and Milestones (POA&M).
